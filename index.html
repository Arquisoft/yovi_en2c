1. Introduction and Goals

We are a team of four Software Engineering students developing YOVI, a web-based gaming platform for the company Micrati as part of our Architecture of Software Systems (ASR) course project.

The platform will allow users to play various versions of Game Y against AI opponents with different strategies, track their performance, and compete through an optional public ranking system. Our goal is to deliver an engaging, intuitive, and reliable gaming experience that can be extended with new game variants in the future.

This document outlines the architectural vision for YOVI, balancing functionality with clarity and maintainability to meet both academic requirements and player expectations.

1.1. Requirements Overview

The YOVI system is a web-based gaming platform based on Game Y, developed for the company Micrati. The main goal is to allow users to play matches against the machine or via bots, with support for multiple strategies, variable board sizes, and registration and statistics functionalities.

Key functional requirements include:

  • Development of a web application in TypeScript that allows playing at least the classic version of Game Y.

  • Implementation of a Rust module to verify the game state and suggest moves, with JSON communication using YEN.

  • A documented public API to allow bot interaction.

  • User registration and access to match history, statistics, and metrics.

  • Support for multiple strategies and difficulty levels in the vs. machine mode.

  • Public deployment and web accessibility.

1.2. Quality Goals

The main quality attributes for the system architecture are:

  • Functionality: The system must support multiple Game Y variants and allow matches against AI with different strategies.

  • Usability: The interface must be intuitive and optionally support internationalization (i18n).

  • Maintainability: The architecture must facilitate the extension with new game variants without rewriting core components.

  • Availability: The application must be deployed and publicly accessible with high availability.

  • Interoperability: The API must be well-documented and allow integration with external bots via YEN.

2. Architecture Constraints

This section lists the main constraints that shape the architecture of the YOVI system. These constraints come from the assignment requirements and limit the available architectural choices.

2.1. Technical Constraints

  • The system is composed of at least two subsystems:

    • A web application implemented in TypeScript.

    • A game engine module implemented in Rust.

  • The Rust module must be exposed as a web service so it can be invoked remotely by the web application.

  • Communication between the web application and the Rust module must use JSON messages following the YEN notation for representing the state of a Game Y match.

  • The web application must expose a public API that allows external bots to interact with the system.

  • The complete system must be deployed and publicly accessible via the Web.

2.2. Domain Constraints

  • The system must implement the rules of Game Y. Support for the classic version of the game is mandatory.

  • A player-versus-machine game mode must be available.

  • The game must support a variable board size.

  • The computer player must implement more than one strategy. The user must be able to select the strategy, optionally including different difficulty levels.

2.3. Organizational and Process Constraints

  • The architecture and design decisions must be documented following the guidelines used in the course, including the use of Architectural Decision Records (ADRs).

  • The system must include automated testing at different levels, such as unit, integration, and end-to-end tests.

  • Development must be carried out using a version-controlled repository, including issue tracking and documented architectural decisions.

  • The project should support automated build and deployment through CI/CD pipelines.

2.4. External Constraints

  • External bots can only interact with the system through the exposed API and must not depend on internal implementation details.

  • The public API must be documented to allow third-party developers to implement compatible bots.

These constraints define the boundaries within which the YOVI architecture has been designed.

3. Context and Scope

This section describes the context of the YOVI system, its external interfaces, and the boundaries between the system and its environment.

3.1. Business Context

YOVI is a web-based platform that allows users to play Game Y against the computer and enables external bots to interact with the system through a public API. The system is intended to be accessed through a web browser or programmatically via HTTP-based interfaces.

The main external actors are:

  • Human players, who access the system through a web frontend.

  • Bots, who interact with the system using the exposed API.

The YOVI system depends on a separate game engine module, implemented in Rust, which is responsible for validating game states, checking winning conditions, and suggesting the next move for the computer player.

3.1.1. Business Context Diagram

3.2. Technical Context

From a technical point of view, YOVI consists of a web application and an external game engine service communicating over HTTP.

  • The web application provides:

    • A browser-based user interface for human players.

    • A REST-style API for bot interaction and game management.

  • The game engine service:

    • Receives game states encoded in YEN notation.

    • Evaluates whether a game has been won.

    • Computes and returns the next move according to the selected strategy.

All communication between the web application and the game engine uses JSON messages following the YEN notation defined for Game Y.

3.3. Scope

The scope of the YOVI system includes:

  • Playing Game Y through a web interface.

  • Playing against the computer using multiple strategies.

  • User registration and access to basic statistics and match history.

  • A public API that allows bots to retrieve information and play matches.

The following aspects are explicitly out of scope or optional:

  • Support for game variants other than the classic version of Game Y (unless implemented as optional features).

  • Mobile applications.

  • Advanced social features such as chat or video communication.

  • Multiplayer modes involving multiple human players.

4. Solution Strategy

The architecture of the application will be designed designed to ensure responsive gameplay, maintainable logic, and high usability, while remaining simple to extend.

4.1. Technology Decisions

This section explains the core technological foundations of the system and why they were selected:

  • Bot Implementation: Heuristic AI to guarantee predictable response times and deterministic behavior which simplifies testing. A trade-off of this choice would be that a heuristic model is less adaptable than a machine learning model and may make less rational decisions in higher dificulties.

  • User Interface: GUI framework to enable clear visualization of game state and give immediate feedback to the player. This will make the experience more enjoyable for the player, however it will be harder to develop than a text-based game.

  • Database: We decided to integrate a Database in order to store user related data. However this will increase system complexity.

  • Internationalization: We will make it so the game can be internationalized. We will make it in English and Spanish, but we will code it in such a way any language or region could be easily implemented. This will require string management from the very beginning and a more complex UI.

4.2. Organizational and Process Decisions

This section explains the way the team organizes itself to ensure quality:

  • Development Process: Iterative and incremental to enable early validation of gameplay and bot behavior. This will require planning of every stage of development.

  • Responsibility Separation: Different layers or microserviecs can be assigned to different developers. This will increase modularity, encourage parallel programming and reduces coupling, however it requires more communication and coordination between developers.

  • Kanban: We will use it as a tool to organize the task and have a clear visualization of the work relative to the rest of the team. It will require continuous updating.

4.3. Architectural Decomposition

We will use Model-View-Controller Architecture based on the implementation of different microservices:

  • Presentation Layer: Handles visualization.

  • Game Logic Layer: Enforces rules and manages game state.

  • Bot AI Layer: Computes bot decisions independently of UI.

  • Infrastructure Layer: Logging, configuration, error handling user interaction.

  • Database Layer: Stores all the relevant data of the application.

4.4. Design Patterns

Our main goal will be to create a game that can be easily modified and easily adapts to future changes in the code, for example, that allows new features or rules into the game. That’s why we chose that our project follow these design patterns:

  • ModelViewController (MVC): Ensures clear separation between game state, user interface, and control logic.

  • Strategy Pattern for bot behavior: Allows multiple bot difficulty levels and easy extension without modifying core logic.

  • Observer Pattern for UI updates: Ensures efficient and consistent propagation of game state changes.

4.5. Quality Goal Realization

This section contains the main quality goals that our project aspires for:

  • Performance: The game must respond in time to player actions.

  • Usability: UI should be intuitive and players should be able to tell what to do either by the UI or by rules explaining the game.

  • Scalability: The game must be able to accept changes in the way of playing or accept new features added in the future.

  • Maintainability: The code should be easy to modify or extend and should have the proper comments and be structured to make it readable.

  • Reliability: Game can run and tackle exceptions without crashing.

  • Portability: The game should run on multiple platforms.

4.6. Conclusion

Our solution strategy deliberately prioritizes modularity, predictability, and extensibility.

Motivation

The architecture is decomposed based on the core technical requirement to separate the user interface and game management logic (suited for TypeScript/Web) from the core game rule verification and AI strategy computation (suited for Rust). This separation aligns with the specified subsystems and promotes modularity, independent scalability, and clear responsibility boundaries.

Contained Building Blocks

  • yovi_web_app: The main web application that provides the user interface, manages game sessions, and handles user data.

  • yovi_game_engine: A standalone service responsible for validating game states, determining winners, and calculating AI moves.

Important Interfaces

  • Game Management API: The RESTful interface exposed by the yovi_web_app to the yovi_game_engine to request move validation and AI suggestions. This interface uses YEN notation over JSON.

  • Public Bot API: The external RESTful API exposed by the yovi_web_app to allow programmatic bots to play the game.

5.1.1. yovi_web_app

  • Purpose/Responsibility: Serves as the primary interface for human users and the gateway for external bots. It is responsible for user management (registration, history), session state, rendering the game UI, and orchestrating calls to the game engine.

  • Interface(s):

  • Provided: Public REST API (/api/v1/*) for bots, Web Interface (HTML/JS) for users.

  • Required: Consumes the Game Management API from the yovi_game_engine.

  • Quality/Performance Characteristics: Must be highly available and responsive for a good user experience. Stateless to allow horizontal scaling.

  • Directory/File Location: /web-app/ (Contains React/TypeScript source code).

  • Fulfilled Requirements: User registration/history, web frontend, public bot API, strategy selection.

  • Open Issues/Problems/Risks: Complexity in managing real-time updates for potential multiplayer features. See technical risks (Section 11).

5.1.2. yovi_game_engine

  • Purpose/Responsibility: Encapsulates the definitive rules of Game Y and its variants. Its core functions are to validate moves, check for a winning condition, and calculate the next move for the AI based on a selected strategy.

  • Interface(s):

  • Provided: Game Management API (Web Service, e.g., over HTTP). Accepts and returns game states in YEN notation.

  • Required: None (it’s a terminal building block for core logic).

  • Quality/Performance Characteristics: Must have low latency for AI calculations to not hinder gameplay. High correctness is critical.

  • Directory/File Location: /game-engine/ (Contains Rust source code, Cargo.toml).

  • Fulfilled Requirements: Game rule verification, AI move suggestion with multiple strategies.

  • Open Issues/Problems/Risks: Performance of complex AI strategies on large boards. Learning curve for Rust (See ADR 3 and Section 11).

The web application is decomposed into four main components:

  • API Client & State Manager: Manages the application state (e.g., current game, user info) and handles all communication with the backend Game Management API. Transforms data between frontend models and YEN notation.

The game engine is decomposed into three logical components:

  • YEN Parser & Validator: Responsible for parsing incoming YEN JSON, validating its syntactic and semantic correctness against the current board rules, and serializing outgoing YEN states.

  • Game Rules Core: The single source of truth for all Game Y variants. Contains the logic for board representation, move application, and win condition checking.

  • AI Strategy Module: Implements the pluggable strategy pattern (aligned with ADR 4). Contains different algorithms (e.g., Minimax, Random) and difficulty levels that can be selected via the API request.

6. Runtime View

6.1. Runtime Scenario 1: Playthgough of a Game against two players

The following diagram shows the flow of a normal playthrough of a game with two players going against eachother :

As such the standard gameplay would be as follows:

Initialization:

  1. User1 navigates to game initialization.

  2. The backend creates a new game session and stores it.

  3. User2 joins the session.

  4. The backend confirms that both players are connected.

  5. The game loop is initialized.

  6. Both users see the initialized game state.

Turns:

  1. The active player makes a move.

  2. The move is sent to the backend.

  3. The backend validates and updates the game state.

  4. The updated state is sent to the opponent.

  5. The opponents GUI renders the updated game state.

  6. The turn switches to the other player.

Finalization:

  1. The backend stores the result of the game in the database (User1 victory or User2 victory).

  2. The backend sends the final result to both GUIs.

  3. The winner sees a victory screen.

  4. The loser sees a defeat screen.

  5. The game session is closed and resources are released.

6.2. Runtime Scenario 2: Playthrough of a Game against a Bot

The following diagram shows the flow of a normal playthrough of a game against a bot (initialization, turns, and finalization):

As such, a standard playthrough would be:

  1. A user navigates to the game initialization

  2. The backend initializes a bot that is returned, therefore initializing the game

  3. The main loop of the game is initialized

Then the loop of the turns would be:

  1. The user makes a move

  2. The move is returned to the backend

  3. A bot move is requested

  4. The move of the bot is done and send to the GUI

It will continue until the game ends, and once that happens:

  1. The result of the game is stored in the database (either user wins or user loses)

  2. The backend sends a signal to the GUI with the result of the game

  3. Depending on the result, either a victory or a defeated screen is shown (which could be the same screen with different information, but this is a design decision)

  4. The bot is released

7. Deployment View

Motivation

The application is built to use Docker and a Virtual Machine as a host in such a way that it combines the portability of the containers with the security and isolation features of the VM.

Quality and/or Performance Features

Some of the characteristics obtained by the use of the specified implementation would be:

  • Isolation: Both thanks to the container isolation of Docker and the own security features of the Virtual Machine.

  • Portability and Compatibility: Thanks to the use of the Docker containers.

  • Security: If a container is compromised, the VM can act as another security layer.

  • Others: Scalability, easily maintainable, would be very simple to backup…​

Mapping of Building Blocks to Infrastructure

The building blocks of the current infrastructure would be:

  • Webapp: Located in port 80, it is the one the user will connect to and interact with. It is the GUI (Graphic User Interface), and as such the frontend of the aplication.

  • Gateway: This block has port 8080 assigned, and it is in charge of acting as a bridge between the rest of the building blocks of the application,

  • Users: It is in charge of the user management as well as the authentication feature, and its port is 3000.

  • Gamey: This service, writen in Rust, is the one containing the complete functionality of the Y Game. It uses port 4000.

  • Database: It is an external block containing all the information of the application (user data, game data…​).

Additionally, it is worth mentioning the presence of Prometheus and Grafana, located in ports 9090 and 9091 respectively, are external blocks that will be used to monitor and obtain metrics of the application and its usage.

9. Architecture Decisions

This section documents the most significant architectural decisions made during the design and development of the YOVI platform. Each decision follows the ADR (Architecture Decision Record) format to provide context, rationale, and consequences.

9.1. 1: Prioritize Early Internationalization (i18n)

  • Status: Accepted

  • Context: One of the optional features for the YOVI platform is support for internationalization (i18n). The team recognized that implementing i18n later in the development cycle is often more complex and costly.

  • Decision: Internationalization will be integrated into the project’s foundation from the start. This includes structuring the codebase for easy text extraction, using a library, and designing the UI layout to accommodate variable text lengths.

  • Consequences:

  • Positive: Significantly reduces future integration effort, avoids major refactoring of the UI, and allows the optional feature to be completed with minimal friction.

  • Negative: Adds initial setup complexity and requires discipline from all developers to use the i18n framework for all user-facing text from day one.

11. Risks and Technical Debts

This section identifies potential technical risks and anticipated technical debts for the YOVI project. Items are ordered by their estimated impact and probability. For each item, mitigation strategies are proposed.

Risk / Technical Debt Potential Impact Mitigation Strategy

Learning Curve for Rust

High

Dedicated learning sprint at project start. Use paired programming and focus on implementing the core game verification module first with extensive testing.

Integration Complexity between TypeScript and Rust

High

Define a clear, versioned JSON interface using YEN notation early. Create integration tests and contract tests to ensure both modules adhere to the agreed protocol.

Incomplete Understanding of Game Y Rules and Variants

Medium-High

Conduct a formal requirements analysis session focused solely on the game domain. Create a shared glossary and develop a "walking skeleton" to validate the core logic early.

Scope Creep from Optional Features

Medium

Prioritize the implementation of mandatory features first. Define clear "Definition of Done" criteria for each optional feature and only address them after the core system is stable.

Technical Debt from Rushed Frontend Decisions

Medium

Adopt an established component library to ensure consistency and responsiveness from the start. Enforce code reviews focused on UI structure.

Lack of Performance Benchmarking for AI Strategies

Medium-Low

Define performance budgets for AI move calculation. Profile the Rust module early with different board sizes and algorithm complexities.

API Stability for External Bots

Medium

Version the public API from the first release. Provide comprehensive and maintained documentation. Consider implementing API deprecation policies early.

Deployment and DevOps Complexity

Low-Medium

Use containerization (Docker) from the beginning to ensure environment parity. Automate the deployment pipeline (CI/CD) early to avoid manual deployment "debt."

12. Glossary

Term Definition

YOVI

Name of the system developed in this project. A web-based platform for playing Game Y, supporting human players and external bots.

Game Y

An abstract strategy board game in which players aim to connect three different sides of the board.

Classic Game Y

The standard version of Game Y defined by the original rules, without additional variants or optional mechanics.

YEN Notation

A JSON-based notation used to represent the state of a Game Y match, including board size, current turn, players, and board layout.

Game State

A complete representation of a Game Y match at a specific point in time, encoded using YEN notation.

TypeScript

A statically typed programming language that extends JavaScript and is used to implement the web application.

Rust

A systems programming language focused on safety and performance, used to implement the game engine module.

API

A public HTTP-based interface exposed by the web application that allows external clients and bots to interact with the system.

Bot

An external program that interacts with the system exclusively through the public API to retrieve information and play games automatically.

Player-versus-Machine

A game mode in which a human player competes against a computer-controlled opponent.

JSON

A lightweight data-interchange format used for communication between the web application and the Rust game engine.

HTTP

The application-layer protocol used for communication between clients, bots, and system services.

Microservice

An architectural style in which a system is composed of small, independent services that communicate over a network and can be developed, deployed, and scaled separately.
