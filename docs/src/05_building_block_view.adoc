== Building Block View

This section describes the static decomposition of the YOVI system into building blocks. Following arc42 guidelines,
we present a **hierarchical view**:

* **Level 1**: White box description of the overall system, with black box descriptions of each top-level building block
* **Level 2**: White box descriptions of selected building blocks, showing their internal structure

This abstraction allows understanding the system structure without drowning in implementation details.

=== Level 1: Overall System White Box

The YOVI system is decomposed into **five top-level building blocks** plus an external database.

==== Black Box Descriptions

===== API Gateway

* **Purpose/Responsibility**: Acts as the single entry point to the system. Routes external requests to the appropriate internal service, handles CORS centrally, and provides a security layer.
* **Interface(s)**:
** *Provided*:
*** Public Web UI endpoint
*** Public Bot API endpoint
** *Required*:
*** Web Frontend Service (internal HTTP)
*** Authentication Service (internal HTTP)
*** Game Engine Service (internal HTTP)
* **Quality/Performance Characteristics**: Must handle all external traffic; availability is critical. Stateless, can be scaled horizontally.
* **Directory/File Location**: `/gateway/gateway-service.js`
* **Fulfilled Requirements**: Public deployment, single entry point, centralized CORS
* **Open Issues**: SSL certificate management in production

===== Web Frontend Service

* **Purpose/Responsibility**: Provides the user interface for human players. Handles game rendering, user interaction, language switching, and orchestrates calls to backend services via the gateway.
* **Interface(s)**:
** *Provided*: Web UI (HTML/CSS/JS) served to browsers
** *Required*:
*** Authentication Service API (via gateway)
*** Game Engine API (via gateway)
* **Quality/Performance Characteristics**: Responsive UI (< 100ms for interactions). Supports EN/ES via custom i18n module. Stateless for scalability.
* **Directory/File Location**: `/webapp/` (React + TypeScript source)
* **Fulfilled Requirements**: Web frontend, game visualization, strategy selection UI, internationalization
* **Open Issues**: Real-time updates for future multiplayer features

===== Authentication Service

* **Purpose/Responsibility**: Handles all identity-related operations: JWT token generation and validation, registration payload validation, and login credential verification. Delegates user data persistence to the User Service.
* **Interface(s)**:
** *Provided*: REST API with endpoints `/register`, `/login`, `/verify`, `/health`
** *Required*: User Service (internal HTTP — user lookup and creation)
* **Quality/Performance Characteristics**: Response time < 200ms. Stateless (JWT-based). Token expiration configurable via `JWT_EXPIRES` env var.
* **Directory/File Location**: `/authentication/auth-service.js`
* **Fulfilled Requirements**: User authentication, JWT generation and validation
* **Open Issues**: Password comparison currently in plain text — bcrypt hashing pending (tracked as TD-5)

===== User Service

* **Purpose/Responsibility**: Manages all user data persistence: user profiles, game results, match history, and statistics. Acts as a pure data store — no authentication logic.
* **Interface(s)**:
** *Provided*: REST API with endpoints `/createuser`, `/users/:username`, `/users`, `/gameresult`, `/history/:username`, `/ranking`, `/health`
** *Required*: MongoDB for data persistence
* **Quality/Performance Characteristics**: Must persist data reliably. Response time < 200ms for typical operations.
* **Directory/File Location**: `/users/users-service.js`
* **Fulfilled Requirements**: User registration (data), match history, statistics, ranking
* **Open Issues**: `/profile/:id` and `/:id/stats` endpoints pending implementation

===== Game Engine Service

* **Purpose/Responsibility**: Encapsulates all Game Y logic: move validation, win condition checking (connecting three sides of the triangular board), and AI move calculation with six strategies.
* **Interface(s)**:
** *Provided*: REST API using YEN notation — `POST /game/new`, `POST /v1/game/pvb/{bot_id}`, `POST /v1/ybot/choose/{bot_id}`, `GET /status`
** *Required*: None (stateless, no external dependencies)
* **Quality/Performance Characteristics**: Low latency (< 2s for move calculation per QS-01). High correctness (100% rule compliance). Memory safety guaranteed by Rust.
* **Directory/File Location**: `/gamey/` (Rust + Axum)
* **Fulfilled Requirements**: Game rule verification, AI move suggestion, six bot strategies, YEN notation, public bot API
* **Open Issues**: Performance optimization for large boards with complex bots

===== MongoDB Database

* **Purpose/Responsibility**: Persists user data, including profiles, authentication credentials, game results, match history, and statistics.
* **Interface(s)**:
** *Provided*: MongoDB Wire Protocol
** *Required*: None (data storage only)
* **Quality/Performance Characteristics**: Durable storage, backup capability. Must handle concurrent connections.
* **Directory/File Location**: Managed via Docker (not in repo)
* **Fulfilled Requirements**: User data persistence, game results, ranking aggregation
* **Open Issues**: Backup and recovery strategy

===== Black Box Descriptions (Webapp Internals)

|===
| Building Block | Responsibility | Key Interfaces | Location
| **UI Components** | Render all user interfaces; handle user input events | Props/State callbacks | `webapp/src/`
| **API Client** | Communicate with backend services via gateway | HTTP fetch calls to `/api/*` | `webapp/src/` (inline in components)
| **Routing** | Handle navigation between views | URL paths (`/`, `/register`, `/home`, `/game`, `/select-difficulty`, `/game/finished`) | `webapp/src/App.tsx`
| **i18n Module** | Provide EN/ES internationalization via React Context | `useI18n()` hook → `t(key)` function | `webapp/src/i18n/`
|===

The game engine is structured in layers, with clear separation between parsing, core logic, and AI strategies.

  HTTP --> Registry : "find bot by name"
  Registry --> AI : "returns YBot impl"

  note right of AI
    **Strategies (YBot trait):**
    * RandomBot
    * HeuristicBot
    * MinimaxBot (depth 3)
    * AlfaBetaBot
    * MonteCarloBot (Hard)
    * MonteCarloBot (Extreme)
  end note
}

@enduml

===== Black Box Descriptions (Game Engine Internals)

|===
| Building Block | Responsibility | Key Interfaces | Location
| **HTTP Server** | Expose REST endpoints via Axum; handle requests/responses | `POST /game/new`, `POST /v1/game/pvb/{bot_id}`, `POST /v1/ybot/choose/{bot_id}` | `gamey/src/game_server/`
| **YEN Parser** | Parse and validate YEN notation; serialize game state to YEN | `GameY::try_from(yen)`, `YEN::from(&game)` | `gamey/src/notation/yen.rs`
| **Game Core** | Implement Game Y rules; validate moves; check win conditions (3 sides connected) | `add_move()`, `available_cells()`, `get_player_positions_coords()` | `gamey/src/core/`
| **AI Strategies** | Implement different bot behaviors via `YBot` trait | `choose_move(&GameY) -> Option<Coordinates>` | `gamey/src/bot/bot_implementations/`
| **Bot Registry** | Store and retrieve bot implementations by name | `find(name) -> Option<Arc<dyn YBot>>` | `gamey/src/bot/ybot_registry.rs`
|===

=== Interface Overview

This table summarizes the key interfaces between building blocks, showing the **provided** and **required** relationships.

|===
| Interface | Provider | Consumer | Protocol | Data Format | Purpose
| **Public Web UI** | Gateway | Browser | HTTPS | HTML/CSS/JS | Serve frontend to users
| **Public Bot API** | Gateway | External Bot | HTTPS | JSON (YEN) | Bot integration
| **Auth API** | Auth Service | Gateway | HTTP (internal) | JSON + JWT | Register, login, verify token
| **User Data API** | Users Service | Auth Service | HTTP (internal) | JSON | User lookup and creation
| **Game API** | Gamey Service | Gateway | HTTP (internal) | JSON (YEN) | Game logic and bot moves
| **Database Interface** | MongoDB | Users Service | MongoDB Wire | BSON | Data persistence
|===

=== Mapping to Requirements

This table shows how the building blocks fulfill the key requirements from section 1.

|===
| Requirement | Primary Building Block | Supporting Building Blocks
| Web frontend (TypeScript/React) | Web Frontend Service | Gateway
| Rust game engine | Game Engine Service | (none)
| JSON + YEN communication | Game Engine Service | Webapp (client), Gateway (routing)
| Public API for bots | Gateway | Game Engine Service
| User registration | Authentication Service | User Service, MongoDB
| User authentication (JWT) | Authentication Service | (none)
| Match history | User Service | MongoDB, Webapp (UI)
| Multiple AI strategies | Game Engine Service | (none)
| Internationalization | Web Frontend Service | (none — custom i18n module)
| Docker deployment | All services | docker-compose
| CI/CD | GitHub Actions | All services
|===
