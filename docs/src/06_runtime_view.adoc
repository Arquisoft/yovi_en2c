[[section-runtime-view]]
== Runtime View

This section describes the **dynamic behavior** of the YOVI system through a representative selection of
architecturally significant scenarios. These scenarios demonstrate how the building blocks defined in Section 5
collaborate at runtime to deliver the required functionality.

=== Runtime Scenario 1: User Registration

A new user creates an account. The authentication service validates credentials, hashes the password with bcrypt,
delegates user creation to the users service, and returns a JWT token with role claims.

[plantuml,"User Registration Sequence",png]
----
@startuml
actor "User" as User
participant "Webapp" as WA
participant "Gateway" as GW
participant "Auth Service" as Auth
participant "Users Service" as Users
database "MongoDB" as DB

User -> WA: Fill registration form\n(username, password, repeatPassword, email)
WA -> WA: Client-side validation
WA -> GW: POST /api/register\n{username, password, repeatPassword, email}
GW -> Auth: POST /register (forward)

Auth -> Auth: Validate payload\n(required fields, password length >= 4,\npasswords match, email format)

alt Validation error
    Auth --> GW: 400 {error: "Password must have at least 4 characters"}
    GW --> WA: Forward error
    WA --> User: Show error message
else Valid payload
    Auth -> Users: POST /createuser\n{username, email, password}
    Users -> Users: bcrypt.hash(password, 10)
    Users -> DB: INSERT User document\n(username, hashedPassword, role: "user")
    Users -> DB: INSERT Notification\n(type: "welcome", recipient: username)

    alt Username already exists (11000)
        DB --> Users: Duplicate key error
        Users --> Auth: 409 {error: "Username already exists"}
        Auth --> GW: 409
        GW --> WA: Forward error
        WA --> User: Show error
    else Created successfully
        DB --> Users: Confirm
        Users --> Auth: 201 {user: {id, username, email, createdAt}}
        Auth -> Auth: jwt.sign({id, username, email, role: "user"}, JWT_SECRET)
        Auth --> GW: 201 {success, token, user}
        GW --> WA: Forward response
        WA -> WA: localStorage.setItem("token", token)\nlocalStorage.setItem("username", username)
        WA --> User: Redirect to /home
    end
end
@enduml
----

=== Runtime Scenario 2: User Login and Session Verification

A returning user logs in. The auth service fetches the stored bcrypt hash from the users service, compares it,
and issues a JWT. The home page verifies the token on every load.

[plantuml,"Login and Verification Sequence",png]
----
@startuml
actor "User" as User
participant "Webapp" as WA
participant "Gateway" as GW
participant "Auth Service" as Auth
participant "Users Service" as Users

User -> WA: Enter username + password
WA -> GW: POST /api/login\n{username, password}
GW -> Auth: POST /login (forward)

Auth -> Users: GET /users/:username
Users --> Auth: 200 {user: {id, username, password: "$2b$...", role, ...}}

Auth -> Auth: bcrypt.compare(password, user.password)

alt Wrong password
    Auth --> GW: 401 {error: "Invalid credentials"}
    GW --> WA: Forward
    WA --> User: Show error
else Correct password
    Auth -> Auth: jwt.sign({id, username, email, role}, JWT_SECRET, {expiresIn: JWT_EXPIRES})
    Auth --> GW: 200 {success, token, user}
    GW --> WA: Forward
    WA -> WA: Store token + username in localStorage
    WA --> User: Redirect to /home

    note over WA: On every protected page load:
    WA -> GW: GET /api/verify\nAuthorization: Bearer <token>
    GW -> Auth: GET /verify (forward)
    Auth -> Auth: jwt.verify(token, JWT_SECRET)

    alt Token expired or invalid
        Auth --> GW: 401 Unauthorized
        GW --> WA: Forward
        WA -> WA: Clear localStorage
        WA --> User: Redirect to / (login)
    else Valid token
        Auth --> GW: 200 {success, user: {id, username, role}}
        GW --> WA: Forward
        WA --> User: Render protected page
    end
end
@enduml
----

=== Runtime Scenario 3: Player vs Machine Game (PvM)

A complete game session between a human player and the AI bot. Demonstrates the game loop, YEN notation
exchange, and result persistence.

[plantuml,"PvM Game Sequence",png]
----
@startuml
actor "User" as User
participant "Webapp" as WA
participant "Gateway" as GW
participant "Gamey" as Gamey
participant "Users Service" as Users
database "MongoDB" as DB

== Initialization ==
User -> WA: Select bot difficulty + board size
WA -> GW: POST /api/game/new {size: N}
GW -> Gamey: POST /game/new {size: N}
Gamey -> Gamey: Create empty board\n(YEN with all "." cells)
Gamey --> GW: YEN {size, turn: 0, players: ["B","R"], layout}
GW --> WA: {ok: true, yen}
WA --> User: Render empty board

== Game Loop ==
loop Until game ends
    User -> WA: Click cell (row, col)
    WA -> GW: POST /api/game/pvb/move\n{yen, bot: "minimax_bot", row, col}
    GW -> Gamey: POST /v1/game/pvb/minimax_bot\n{yen, row, col}

    Gamey -> Gamey: Validate + apply player move\n(Union-Find win check)

    alt Player won
        Gamey --> GW: {yen, finished: true, winner: "B", winning_edges}
        GW --> WA: Forward
        WA -> GW: POST /api/gameresult\n{username, opponent: "minimax_bot", result: "win", ...}
        GW -> Users: POST /gameresults
        Users -> DB: INSERT GameResult
        WA --> User: Navigate to /game/finished (victory)
    else Game continues
        Gamey -> Gamey: Calculate bot move\n(MinimaxBot.choose_move)
        Gamey -> Gamey: Apply bot move\n(Union-Find win check)

        alt Bot won
            Gamey --> GW: {yen, finished: true, winner: "R", winning_edges}
            GW --> WA: Forward
            WA -> GW: POST /api/gameresult\n{result: "loss"}
            GW -> Users: POST /gameresults
            Users -> DB: INSERT GameResult
            WA --> User: Navigate to /game/finished (defeat)
        else Game continues
            Gamey --> GW: {yen, finished: false}
            GW --> WA: Forward
            WA --> User: Render updated board
        end
    end
end
@enduml
----

=== Runtime Scenario 4: Real-Time Multiplayer (PvP via Socket.IO)

Two human players compete in real time. Demonstrates room creation, Socket.IO event flow, and move
synchronization between clients.

[plantuml,"PvP Multiplayer Sequence",png]
----
@startuml
actor "Player 1 (Host)" as P1
actor "Player 2 (Guest)" as P2
participant "Webapp (P1)" as WA1
participant "Webapp (P2)" as WA2
participant "Gateway" as GW
participant "Multiplayer\nService" as Multi
participant "Gamey" as Gamey

== Room Creation (Player 1) ==
P1 -> WA1: Click "Create Room" (size 7)
WA1 -> GW: POST /api/multiplayer/room/create\n{username: "P1", size: 7}
GW -> Multi: POST /rooms/create
Multi -> Gamey: POST /game/new {size: 7}
Gamey --> Multi: Initial YEN
Multi -> Multi: Generate room code "ABCD12"\nAssign P1 as color "B"
Multi --> GW: {ok, room: {code, yen, status: "waiting"}, yourColor: "B"}
GW --> WA1: Forward
WA1 --> P1: Show room code "ABCD12"\nConnect Socket.IO
WA1 -> Multi: socket.emit("join_room", {code: "ABCD12", username: "P1"})
Multi --> WA1: socket.emit("room_updated", {room})

== Room Join (Player 2) ==
P2 -> WA2: Enter code "ABCD12" + click "Join"
WA2 -> GW: POST /api/multiplayer/room/join\n{code: "ABCD12", username: "P2"}
GW -> Multi: POST /rooms/join
Multi -> Multi: Assign P2 as color "R"\nStatus -> "active"
Multi --> GW: {ok, room, yourColor: "R"}
GW --> WA2: Forward
WA2 -> Multi: socket.emit("join_room", {code: "ABCD12", username: "P2"})
Multi --> WA1: socket.emit("game_started", {room, message: "Both players connected"})
Multi --> WA2: socket.emit("game_started", {room})
WA1 --> P1: Render board (P1 turn, color B)
WA2 --> P2: Render board (waiting for P1)

== Turn Loop ==
loop Until game ends
    P1 -> WA1: Click cell (row=0, col=0)
    WA1 -> Multi: socket.emit("make_move", {code, row: 0, col: 0})
    Multi -> Multi: Validate it is P1 turn (color "B")
    Multi -> Gamey: POST /v1/game/pvp/move\n{yen, row: 0, col: 0}
    Gamey --> Multi: {yen: updated, finished, winner, winning_edges}
    Multi -> Multi: Update room YEN

    alt Game continues
        Multi --> WA1: socket.emit("game_updated", {room, finished: false})
        Multi --> WA2: socket.emit("game_updated", {room, finished: false})
        WA1 --> P1: Update board (now P2 turn)
        WA2 --> P2: Update board (P2 turn)
    else Game over
        Multi --> WA1: socket.emit("game_updated", {...})\nsocket.emit("game_over", {room, winner})
        Multi --> WA2: socket.emit("game_updated", {...})\nsocket.emit("game_over", {room, winner})
        WA1 --> P1: Show result screen
        WA2 --> P2: Show result screen
    end
end
@enduml

=== Runtime Scenario 5: Friend Request and Notification Flow

A user searches for another user, sends a friend request, and the recipient receives and accepts it.

[plantuml,"Friend Request Sequence",png]
----
@startuml
actor "User A" as UA
actor "User B" as UB
participant "Webapp (A)" as WA
participant "Webapp (B)" as WB
participant "Gateway" as GW
participant "Users Service" as Users
database "MongoDB" as DB

== User Search ==
UA -> WA: Type "bob" in search box
WA -> GW: GET /api/search?q=bob\n(debounced 400ms)
GW -> Users: GET /search?q=bob
Users -> DB: Query users where username or realName contains "bob"
DB --> Users: [User B profile]
Users --> GW: {success, users: [{username: "bob", ...}]}
GW --> WA: Forward
WA --> UA: Show User B card

== Friend Request ==
UA -> WA: Click "Add friend" on User B card
WA -> GW: POST /api/friends/request/bob\nAuthorization: Bearer <token_A>
GW -> GW: sanitizeAuthHeader(token)\nvalidateUsernameParam("bob")
GW -> Users: POST /users/bob/friends/request\n{from: "alice"}
Users -> DB: $push {friendRequests: "alice"} on User B
Users -> DB: INSERT Notification\n{recipient: "bob", type: "friend_request", from: "alice"}
DB --> Users: Confirm
Users --> GW: 201 {success, message: "Friend request sent"}
GW --> WA: Forward
WA --> UA: Button shows "Request sent"

== Notification (User B logs in or refreshes) ==
UB -> WB: Navigate to /home
WB -> GW: GET /api/notifications\nAuthorization: Bearer <token_B>
GW -> Users: GET /users/bob/notifications
Users -> DB: Find notifications for bob, sort by createdAt DESC
DB --> Users: [{type: "friend_request", from: "alice", read: false}]
Users --> GW: {success, notifications}
GW --> WB: Forward
WB --> UB: Bell icon shows unread badge (1)

== Accept Friend Request ==
UB -> WB: Open notification panel, click "Accept" on alice's request
WB -> GW: POST /api/friends/accept/alice\nAuthorization: Bearer <token_B>
GW -> Users: POST /users/bob/friends/accept\n{from: "alice"}
Users -> DB: Remove "alice" from bob.friendRequests\n$push "alice" to bob.friends
Users -> DB: $addToSet {friends: "bob"} on alice
DB --> Users: Confirm
Users --> GW: 200 {success}
GW --> WB: Forward
WB --> UB: Notification marked, friend list updated

WB -> GW: PATCH /api/notifications/:id/read\nAuthorization: Bearer <token_B>
GW -> Users: PATCH /users/bob/notifications/:id/read
Users -> DB: {read: true}
DB --> Users: Confirm
Users --> GW: 200 {success, notification}
GW --> WB: Forward
WB --> UB: Bell badge cleared
@enduml

=== Runtime Scenario 6: Admin Panel — Delete User Account

An administrator uses the admin panel to delete another user's account. Demonstrates role enforcement and
cascading data deletion.

[plantuml,"Admin Delete User Sequence",png]
----
@startuml
actor "Admin User" as Admin
participant "Webapp" as WA
participant "Gateway" as GW
participant "Users Service" as Users
database "MongoDB" as DB

Admin -> WA: Navigate to /admin
WA -> WA: Check JWT role claim\n(role === "admin" ?)

alt Not admin
    WA --> Admin: Redirect to /home (403)
else Is admin
    WA -> GW: GET /api/admin/users?page=1\nAuthorization: Bearer <admin_token>
    GW -> GW: Verify JWT role claim = "admin"
    GW -> Users: GET /admin/users?page=1
    Users -> DB: Query all users (paginated)
    DB --> Users: User list
    Users --> GW: {success, users, total}
    GW --> WA: Forward
    WA --> Admin: Show user management table

    Admin -> WA: Click "Delete account" on target user
    WA --> Admin: Confirmation dialog

    Admin -> WA: Confirm deletion
    WA -> GW: DELETE /api/admin/users/targetuser\nAuthorization: Bearer <admin_token>
    GW -> GW: Verify JWT role claim = "admin"
    GW -> Users: DELETE /admin/users/targetuser

    Users -> DB: DELETE User where username = "targetuser"
    Users -> DB: DELETE GameResults where username = "targetuser"\nor opponent = "targetuser"
    Users -> DB: DELETE Notifications where recipient = "targetuser"
    Users -> DB: $pull {friends: "targetuser"} from all users
    DB --> Users: Confirm all deletions
    Users --> GW: 200 {success, message: "User deleted successfully"}
    GW --> WA: Forward
    WA --> Admin: Refresh user list, show confirmation
end
@enduml
----

=== Runtime Scenario 7: External Bot Using the Interoperability API

An external bot creates a game session against one of our bots, plays moves, and receives the updated state.

[plantuml,"BotAPI External Game Sequence",png]
----
@startuml
actor "External Bot" as Bot
participant "Nginx" as Nginx
participant "BotAPI" as BAPI
participant "Gamey" as Gamey

== Create Game ==
Bot -> Nginx: POST /interop/games\n{size: 5, bot_id: "random_bot"}
Nginx -> BAPI: POST /games (forward)
BAPI -> BAPI: Validate input (size > 0, bot_id not empty)
BAPI -> Gamey: POST /game/new {size: 5}
Gamey --> BAPI: Initial YEN
BAPI -> BAPI: Store ActiveGame\n{gameId: uuid, botId, yen, status: "ONGOING"}
BAPI --> Nginx: 201 {game_id, bot_id, position: YEN, status: "ONGOING"}
Nginx --> Bot: Forward response

== Play a Move ==
Bot -> Nginx: POST /interop/games/:gameId/play\n{position: YEN_with_bot_move}
Nginx -> BAPI: POST /games/:gameId/play (forward)
BAPI -> BAPI: getOrThrow(gameId)\nassertValidYen(proposed)
BAPI -> BAPI: detectSingleAddedMove(stored_yen, proposed_yen)\n(validates exactly one new cell)
BAPI -> Gamey: POST /v1/game/pvb/random_bot\n{yen: stored_yen, row, col}
Gamey -> Gamey: Apply player move\nCompute bot move\nCheck win condition
Gamey --> BAPI: {yen: updated, finished, winner}
BAPI -> BAPI: Update stored game\nComputeStatus (ONGOING / BOT_WON / OPPONENT_WON)
BAPI --> Nginx: 200 {game_id, position: updated_YEN, status}
Nginx --> Bot: Forward response

== Query State ==
Bot -> Nginx: GET /interop/games/:gameId
Nginx -> BAPI: GET /games/:gameId (forward)
BAPI -> BAPI: getOrThrow(gameId)
BAPI --> Nginx: 200 {game_id, bot_id, position, status}
Nginx --> Bot: Forward response
@enduml

=== Runtime Scenario 8: Error and Failure Handling

==== Scenario 8.1: Game Engine Unavailable

[plantuml,"Game Engine Down",png]
----
@startuml
actor "User" as User
participant "Webapp" as WA
participant "Gateway" as GW
participant "Gamey" as Gamey

User -> WA: Make move
WA -> GW: POST /api/game/pvb/move
GW -> Gamey: Forward

note over Gamey: Container crashed or unreachable

GW -> GW: Axios connection refused / timeout
GW --> WA: 502 {ok: false, error: "Game server unavailable"}
WA --> User: Show error message\n(user can retry)
@enduml

----
